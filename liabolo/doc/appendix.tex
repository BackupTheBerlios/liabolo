    \subsection{Allgemein}
    Die letzten Unklarheiten vor der Erstellung eines Prototyps sollen beseitigt werden.
    \subsection{Themen}
    \subsubsection{Signaturerzeugung}
    Die Erzeugung der Signatur ist immer noch problematisch, da noch
    nicht alle Informationen aus der Bibliothek verf\"ugbar sind. Eine Liste mit Fachk\"urzeln
    muss noch erstellt werden. Das Schl\"usselwort <DC.subject> soll mit Schl\"usselw\"ortern gef\"ullt werden.
    \subsubsection{Metatag: Standort}
    Das Metatag Standort soll nicht bei jedem Dokument in einer XML Datei gespeichert werden.
    Als Alternative soll eine Tabelle mit Standorten angelegt werden, die erweitert werden kann.
    \subsubsection{Client/Server Architektur vs. Peer to Peer Netzwerk}
    Beschluss: der Client/Server Struktur wird der Vorzug gegeben, da weniger Netzverkehr auf langsam
    angebundene Clients entsteht, und die Datenkonsistenz h\"oher eingesch\"atzt wird. Probleme bei
    gleichzeitigen \"anderungen auf verschiedenen Clients werden durch die zentrale Instanz minimiert.
    \subsubsection{Zuweisung Buch : Fach}
    Festlegung: Die Zuordnung eines Buches zu einem Fachbereich soll eineindeutig sein. Es soll eine
    1:1 Beziehung modelliert werden. Die Fachbereich und ihre Unterabteilungen sollen in einer Baumstruktur
    gespeichert werden. Die Liste mit den Fachbereichen liegt allerdings noch nicht vor. Nur Nutzer, die
    online arbeiten, sollen neue Kategorien anlegen d\"urfen.
    \subsubsection{Gestaltung der grafischen Oberfl\"achen}
    Das urspr\"ungliche Konzept eines Offline Client mit Java Swing Oberfl\"ache und eines Servers mit
    JSP Oberfl\"ache wird verworfen. Stattdessen soll eine GUI mit einem Java Applet entwickelt werden.
    Dadurch muss die Oberfl\"ache nur einmal gestaltet werden. Der Nutzer kann auf Knopfdruck einen
    Abgleich mit dem Server einleiten, ansonsten kann er lokal arbeiten.
    \subsubsection{Prototyp}
    Ein Prototyp soll erstellt werden. Jurij und Maxim erstellen eine GUI mit Java als Applet mit
    folgenden Eigenschaften:
      \begin{itemize}
          \item{Menu:} Das Menu soll die Punkte: Datei, Bearbeiten, Hilfe haben
          \item{Sprachanpassung:} Das gesamte Programm soll in Englisch erstellt werden. Danach ist eine
          Anpassung an die deutsche Sprache geplant.
          \item{Abfragemaske:} Die 15 Dublin Core Elemente sollen abgefragt werden k\"onnen.
          \item{Die eingegebenen Daten sollen zuerst in eine Datei geschrieben werden, sp\"ater in die Datenbank
           (Xindice)}
      \end{itemize}
    \subsubsection{Leistungsmerkmale des Clients}
      \begin{itemize}
          \item{Eingabe eines Datensatzes}
          \item{Ausgabe aller Datens\"atze}
          \item{\"anderung von Datens\"atzen}
          \item{Fachliste/Standortliste editieren}
          \item{Bildung einer eindeutigen Signatur, Besitzer und Standort eines Dokuments soll enthalten sein}
          \item{Schlagwortsuche}
      \end{itemize}

\subsubsection{Bildung der Signatur}
\begin{itemize}
    \item{Drei Kleinbuchstaben werden f\"ur jeden Fachbereich verwendet}
    \item{Algorithmus f\"ur die Generierung eines Fachbereiches muss erstellt werden}
    \item{Die Eingabe muss darauf \"uberpr\"uft werden, ob Fachbereich aus drei kleinen Zeichen besteht}
    \item{Das Ergebnis muss in einer XML Datei gespeichert werden (UTF-8)}
    \item{Der Name und das K\"urzel werden in einer Baumstruktur gespeichert. Ein Knoten enth\"alt jeweils einen Namen und das K\"urzel.

    \item{ Die Signatur muss ausgelesen und dekodiert werden k\"onnen}
\end{itemize}

\subsubsection{Speicherung des Standortes}
Unklar ist noch, ob eine Liste mit Standorten bereit gestellt werden soll und wie. Jeder Client verwaltet lokal seine eigene Datenbank.
Auf dem Server werden alle Datenbanken der Clients gespeichert. Mit der URN k\"onnen Datenbanken und Dokumente eindeutig angesprochen werden.
Jeder Nutzer hat entweder nur Leserechte auf einer Datenbank oder Lese- und Schreibrecht.

\subsection{Format der XML-Dateien}

F\"ur die Fachzuordnung sollen feste Kategorien vorgegeben werden. Schlagworte m\"ussen am Client selber eingegeben werden ('DC.subject'). Musikst\"ucke
sollen ebenfalls aufgenommen werden k\"onnen. Das Format der CDDB Datenbank soll auf Kompatibilit\"at \"uberpr\"upft werden.
Die Felder mit dem Prefix 'UP.' sollen in 'UNI-OL.' umbenannt werden. Das Dublin Core Metatag 'DC.relation' soll nicht verwendet werden.
Das Metatag UNI-OL.edition soll f\"ur die Angabe der Auflage hinzugef\"ugt werden. Das Tag 'UNI-OL.department' soll in 'UNI-OL.branch' umbenannt
werden.
\\
F\"ur jedes Dokument in der Datenbank m\"ussen folgende Tags Pflicht sein: title, creator, subject, publisher, date (Jahr), identifier.\\

\subsubsection{Kategorisierung}

Etwa 30 verschiedene Kategorien sollen per Drop-Down-Menu angew\"ahlt werden k\"onnen. Alle Kategorien m\"ussen erfasst werden k\"onnen.
Da es zuviele Unterkategorien gibt, wird auf Unterkategorien verzichtet. Der Client soll Kategorien anlegen, bearbeiten und l\"oschen k\"onnen.

    \subsection{Allgemein}

     Alle Projektteilnehmer sollen notieren, welche Arbeiten sie durchgef\"uhrt haben, um sp\"atere Dokumentation zu erm\"oglichen.
     Die Use Case sollen besprochen werden. S\"amtliche Datenbanken der Clients sollen in einer Registry verwaltet werden. Ein Dispatcher
     soll dann die Datenbanken mit den Programmfunktionen verbinden. Das Backend \"ubernimmt die Kommunikation mit den Datenbanken.
     Der Client soll als Standalone-Applikation entworfen werden, um u.a. Problemen mit der JAVA Sandbox aus dem Weg zu gehen.


    \subsubsection{Anwendungsfalldiagramm (Use-Case)}
    Beim Start des Clients soll folgende Funktionalit\"at gegeben sein:
    \begin{itemize}
    \item Eine lokale Datenbank wird mitgeliefert und initialisiert
    \item Der Nutzer gibt seine pers\"onlichen Daten ein
    \item Der Nutzer vergibt ein Passwort f\"ur die lokale Datenbank
    \item Es wird immer nur mit einer Datenbank gearbeitet, f\"ur jede DB kann ein Konto erstellt werden
    \item Die Konten m\"ussen bearbeitet werden k\"onnen
    \item Nach dem Einloggen kann in der Datenbank gesucht werden
    \item Eine leere Liste wird bereitgestellt
    \end{itemize}



\subsubsection{Unterpunkt Neu: f\"ur die Eingabe von Dokumenten}
----------------------------------------------
\begin{itemize}
\item Der Nutzer entscheidet, welche Art von Dokument er anlegen will
\item Das passende XML-Schema wird ausgelesen, eine GUI wird generiert
\item Der Nutzer tr\"agt die Daten ein (inkl. Standort)
\item Eine Signatur wird generiert
\item Die URI wird generiert
\item Ein Standort muss angegeben werden.
\item Es gibt eine Kategorie: 'nicht einsortiert'
\item B-Anforderung: Vorgabe von Standorten wird sp\"ater implementiert
\end{itemize}


\subsubsection{Unterpunkt: \"Andern}
Ein Rechtsklick auf ein Dokument oder das Anw\"ahlen des Buttons: '\"Andern' soll die Bearbeitung eines
Eintrags erm\"oglichen. Der Nutzer soll einen Dialog angezeigt bekommen, in dem er die Dokumenteigenschaften ver\"andern kann.

\subsubsection{Unterpunkt: L\"oschen}
Nach einer Sicherheitsabfrage kann ein Eintrag gel\"oscht werden.

\subsubsection{Unterpunkt: Kategorien}
Hinter dem Unterpunkt Kategorien verbirgt sich eine Liste mit vorgegebenen Eintr\"agen. Kategorien k\"onnen angelegt, ge\"andert oder gel\"oscht werden.


\subsubsection{Unterpunkt Schlagwortliste}
Der Unterpunkt Schlagwortliste ist ein Pop-Up Menu mit vorgegebenen Eintr\"agen. Schlagworte k\"onnen angelegt, ge\"andert oder gel\"oscht werden.

\subsection{Individualliste}
Es existiert eine Gesamtliste, die alle Dokumente der Datenbanken enth\"alt. Jedes Element einer individuellen Liste
erh\"alt einen Button, mit dem das Dokument der Gesamtliste hinzugef\"ugt werden kann.

\subsection{Signaturerstellung}
\begin{itemize}
\item Der Nutzer legt ein Dokument an
\item Die Signatur wird generiert
\item Das Dokument wird in der Individualliste und der Gesamtliste gespeichert
\end{itemize}

\subsection{Eingabe von XML- Daten}
\begin{itemize}
\item Die Kategorie wird aus einer Liste ausgew\"ahlt\\
\item Die Signatur wird mt Hilfe der Listeneintr\"age generiert
\item Das Dokument wird in der Gesamt- und Individualliste gespeichert
\item Die Individualliste referenziert die Gesamtliste
\item Der Dateneintrag wird in der lokalen Datenbank gespeichert
\end{itemize}

\subsection{Konten f\"ur eine Datenbank}
Jedes Konto stellt nur eine Verbindung her. Entweder der Server wird kontaktiert, oder ein weiterer Client. F\"ur einen Verbindungsaufbau m\"ussen folgende Daten bekannt sein: URL der Gegenstelle, sowie Name und Passwort. Mehrere Konten k\"onnen mit einem Client verwaltet werden. In der Statusleiste soll das gerade aktive Konto angezeigt werden. Konten k\"onnen mit dem Client angelegt, ge\"andert oder gel\"oscht werden.

\subsection{Server}
Auf dem Server wird die Gesamtdatenbank gespeichert. Die Datenbank muss initialisiert und mit einem Passwort gespeichert werden. Es wird ein Installationsprogramm f\"ur den Server ben\"otigt.

\subsection{Backend}
Das Backend realisiert die Anbindung des Clients an die Datenbank. Verbindungen werden auf- und abgebaut. Weiterhin muss die Suche nach Dokumenten implementiert werden. Die Funktionen Anlegen, \"Andern, L\"oschen sind sowohl f\"ur die Dokumente. als auch f\"ur die Standorte zu programmieren. Der Branch soll eine Liste der m\"oglichen Kategorien enthalten. Das Backend abstrahiert die Datenbank von der GUI.

\subsection{eXist statt Xindice}
Xindice hat sich als zu unflexibel erwiesen. Als Alternative kommt eXist in Betracht. eXist erm\"oglicht die Abbildung einer Rechteverwaltung f\"ur die Dokumente in der Datenbank. Es besteht die M\"oglichkeit, ein Backup anzulegen. Mit der Rechteverwaltung ist es m\"oglich, Lese- und Schreibzugriff getrennt zu vergeben.

\subsection{XML}
Das interne Feld UNI-OL.archivesource soll in die XML-Beschreibung der Dokumente aufgenommen werden. Hier soll eine URL eingegeben werden k\"onnen, um elektronische Dokumente zu referenzieren.

\subsection{Vorschau auf das n\"achste Treffen}

BWL\_00000001\

\subsection{eXist}
Die Datenbank speichert die Datens\"atze nicht im XML Format, der Zugriff erfolgt \"uber eine API.
Die Bibliotheksdatenbank soll Collection heißen. \

Beispiel f\"ur einen Verzeichnisbaum, in dem die Medataen gespeichert werden sollen:\\

\begin{verbatim}
/home/pum/branch\_list

home/nutzer/branch\_list/
                                    inf
                                    bwl
                                    the
                    individual\_list/
                                    inf
                                    bwl
                                    list
\end{verbatim}


Alles unterhalb von branch\_list wird repliziert, allerdings nicht die individual\_list.
Nach dem Verbindungsaufbau zum Server sollen die eigenen Daten auf den Server \"ubertragen werden.
Der Client installiert eine  eigene Datenbank (eXist), diese wird beim ersten Start generiert,
Die branch\_list und die individual\_list werden angelegt, ebenso die Kategorien.

Die Eintr\"age unter  'nutzer' werden mit dem Server repliziert, die individual\_list ist nur f\"ur den privaten
Gebrauch. Wenn eine URI nicht lokal verf\"ugbar ist, dann wird eine Fehlermeldung ausgegeben. Lokal eingegebene Daten werden in der branch\_list gespeichert.

Vorstellung der JAVA-API f\"ur den Zugriff auf die Datenbank

Es wird eine Liste ben\"otigt, mit der die Fachbereiche einer Beschreibung gegen\"uber gestellt werden k\"onnen. (Inf=Informatik)
Bestehende Kategorien k\"onnen vom Server heruntergeladen werden. Dokumente, die keiner Kategorie zugeordnet werden, bekommen
das K\"urzel 'N-A' (not available). Kategorien d\"urfen nicht gel\"oscht werden, damit Signaturen immer referenzieren k\"onnen. Die Beschreibung darf allerdings
ge\"andert werden.

\begin{verbatim}
Bsp:

      <all>

    \section{Projektbezogenes Vorgehen}
        Im Wesentlichen gibt es folgende targets, die von den Projektmitgliedern benutzt werden sollten.\\
        Folgende Aufrufe sollten nach erfolgreicher Ant-Installation und Checkout des Projektes getätigt werden:        
        \begin{enumerate}
            \item{extract\_exist: Extrahiert die Datenbank. Aufruf nur einmal nach dem Checkout nötig!}
            \item{ant start\_exist :Startet die Datenbank}
            \item{ant stop\_exist : Beendet die Datenbank}
            \item{ant p30\_offlineClient : Kompiliert, deployed und startet den Offline-Client}
            \item{ant p30\_projektbericht: ruft pdflatex, bibtex und makeindex in richtigen     
            Intervallen auf}
        \end{enumerate}     
\chapter{Handhabung Versionsmanagement}
    \label{cvs_p30}
    \section{Projektbezogenes Vorgehen}
        Folgende Beschreibung basiert auf der Annahme, daß ein CVS-Konsolen-Client vorliegt.\\
        \subsection{Anwendung für den Benutzer}
            Für den Benutzer sind in der Regel nur Releases interessant. Daher wird auch nur auf ein Checkout eines Releases eingegangen. Falls andere Funktionen erwünscht sind, wird auf den Entwickler-Abschnitt(\ref{cvs_developer}) verwiesen.\\
            Unter Annahme, daß die Bezeichnung des gewünschten Releases zur Verfügung steht, sollten ´folgende Aufrufe genügen :\\
      \begin{Verbatim}[tabsize=2,frame=leftline,label=build.xml,numbers=left]
cvs -d:pserver:anonymous@srv.wiwi.uni-oldenburg.de:/var/lib/cvs login
cvs -d:pserver:anonymous@srv.wiwi.uni-oldenburg.de:/var/lib/cvs checkout
                         -r [release_name] p30
      \end{Verbatim}
        \label{cvs_developer}
        \subsection{Anwendung für den Enwickler}
            Entwickler werden neben dem reinen Checkout eines Projektes auch Änderungen vornehmen. Da hierbei CVS in intensiverem Maße als beim reinen Benutzer benutzt wird, empfiehlt es sich folgende Variable zu setzen.\\
            \textbf{CVSROOT=:pserver:[USER\_NAME]:[PASSWORD]@srv.wiwi.uni-oldenburg.de:/var/lib/cvs}\\
            Von nun an können folgende Kommandos vereinfacht abgegeben werden:\\
            \begin{itemize}
                \item{Einloggen: cvs login}
                \item{HEAD Auschecken: cvs checkout}
                \item{Änderungen einholen: cvs update}
                \item{Änderungen einchecken}
                    \begin{enumerate}
                        \item{cvs update}
                        \item{cvs commit}
                    \end{enumerate}
                \item{Element hinzufügen}
                    \begin{enumerate}
                        \item{cvs add [ELEMENT\_NAME]}
                        \item{cvs commit [ELEMENT\_NAME]}
                    \end{enumerate}
                \item{Element löschen} 
                    \begin{enumerate}
                        \item{Datei lokal löschen: del/rm [ELEMENT\_NAME]}
                        \item{cvs remove [ELEMENT\_NAME]}
                        \item{cvs commit [ELEMENT\_NAME]}
                    \end{enumerate}
            \end{itemize}
            Mit obigen Kommandos sollte wesentliche Funktionaltiät erklärt sein. Bei speziellen Ereignissen wird auf die Literatur verwiesen\cite{cvs_doc}.
